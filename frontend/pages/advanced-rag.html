How It Works
Step 1 — Document Ingestion
Text, tables, and images are extracted separately, preserving document structure. A cleaning step removes footers, encoding issues, and boilerplate. Chunking uses semantic boundaries, sliding window, or parent-child strategies with overlap — no idea is cut mid-thought. Rich metadata including document name, page number, section, and content type is stored alongside every chunk.
Step 2 — Multi-Vector Indexing
Each chunk receives multiple embedding vectors — one for full content, one for a summary, and one for hypothetical questions the chunk can answer. Storage holds all vectors, the raw text, and the complete metadata. This enables retrieval from multiple angles for the same piece of content.
Step 3 — Query Processing & Retrieval
The user's question is first rewritten, decomposed into sub-questions, or expanded using HyDE before retrieval. Hybrid search then combines vector similarity and keyword search, with metadata filtering to narrow the candidate pool. A cross-encoder re-ranker scores each candidate for precise relevance — unlike the bi-encoder used in indexing, the cross-encoder sees both query and chunk together, making it far more accurate.
Step 4 — Post-Processing & Generation
Before the LLM sees anything, chunks are filtered by relevance threshold, deduplicated, compressed to extract only relevant sentences, and reordered so the most relevant content sits at the strongest attention positions in the context window. The prompt is structured with source citations. A final validation step checks the answer for faithfulness and completeness against the retrieved evidence.

Step 2 — Advanced Chunking
Unlike naive fixed-size splitting, Advanced RAG uses strategies that respect meaning boundaries. No idea is cut mid-thought.
Sentence-Aware Chunking
Splits only at sentence boundaries — never in the middle of a sentence. Preserves grammatical completeness.
Semantic Chunking
Groups consecutive sentences by embedding similarity. A new chunk begins when topic drift is detected — keeps related ideas together.
Recursive Chunking (Hierarchical)
Splits top-down: section → paragraph → sentence. Small child chunks are stored for precise retrieval; parent chunks are kept for richer LLM context.
Input text (paragraph):
"Self-attention computes relationships between all tokens in a sequence. Each token attends to every other token, producing a weighted sum of value vectors."

Sentence-aware output (2 chunks):
Chunk A: "Self-attention computes relationships between all tokens in a sequence."
Chunk B: "Each token attends to every other token, producing a weighted sum of value vectors."
Step 3 — Metadata Enrichment
Every chunk is tagged with structured metadata at ingestion time. This metadata powers filtering, ranking boosts, and routing in agentic systems.
Chunk metadata example:
{
  "source": "attention_paper.pdf",
  "section": "3.2 Self-Attention",
  "topic": "Transformers",
  "year": 2017,
  "chunk_id": 42
}
At query time, metadata enables pre-filtering (only search chunks from a specific year or section), ranking boosts (prefer recent sources), and traceability — every answer can be traced to an exact source and section.
Step 4 — Dual Storage: Two Parallel Systems
Each chunk is stored in two completely separate systems — each optimised for a different type of search. These run independently; they do not compare against each other.
Vector Database
Stores the embedding vector for each chunk. Used for cosine / semantic similarity search.
"self attention computes..."
→ [0.21, -0.88, 0.77, ...]
→ stored in Pinecone / Milvus / NumPy
Keyword Index (Inverted Index)
Stores a term → chunk mapping. Lookup is direct — no vector comparison ever happens here.
"attention" → [chunk 3, chunk 17, chunk 42]
"transformer" → [chunk 1, chunk 42]
→ stored in Elasticsearch / TF-IDF matrix
Key distinction: The keyword index is an inverted index — a term-to-document lookup table, not a similarity-scored structure. A keyword query extracts terms and looks them up directly in this table. No embedding or cosine comparison is involved.
Query Time Phase (Online)
Step 5 — Query Understanding Layer
Before retrieval, the raw user question is transformed to maximise retrieval quality. Any combination of these techniques can be applied.
Query Rewriting
An LLM rewrites the raw question to be more retrieval-friendly — more specific, precise, and aligned with how document text is phrased.
Original: "How does attention work?"
Rewritten: "Explain the self-attention mechanism in transformer models and how it computes token relationships"
HyDE — Hypothetical Document Embeddings
An LLM generates a hypothetical answer to the question. That answer (not the original question) is embedded for retrieval. A plausible answer is semantically closer to real stored chunks than a short question is.
Question: "How does self-attention work?"
HyDE answer: "Self-attention computes a weighted sum of values by comparing a query vector..."
→ embed this hypothetical answer → use for vector search
Query Expansion & Multi-Query
Generate multiple paraphrases or sub-questions from one input, run all of them through retrieval, then merge. Improves recall by covering more of the semantic space.
Step 6 — Parallel Retrieval: Two Independent Paths
Vector search and keyword search run simultaneously and completely independently. They do not compare results with each other at this stage.
Vector Search Path
Query is embedded → cosine similarity computed against all stored vectors → top-k by score.

2Chunk B
3Chunk C
Keyword Search Path (Inverted Index)
Keywords extracted → direct lookup in the inverted index → BM25/TF-IDF scored. No vectors involved.

1Chunk C
2Chunk D

Important: No direct comparison between vector and keyword results happens here. Fusion only occurs in the next step. This separation is what allows each retriever to operate at maximum efficiency using its native data structure.
Step 7 — RRF Fusion → Candidate Pool
The two ranked lists are merged using Reciprocal Rank Fusion. Each chunk scores 1/(60+rank) from each list it appears in — scores are summed. No raw similarity scores are compared. This produces a combined candidate pool of 20–100 chunks for reranking.

1Chunk C#3 in vector + #1 in keyword
2Chunk A#1 in vector + #3 in keyword
3Chunk D#2 in keyword only
4Chunk B#2 in vector only
These become the candidate pool passed to the cross-encoder — typically 20–100 chunks before the next precision cut.
Step 8 — Cross-Encoder Reranking
This is the first stage where query and chunk are seen together. The bi-encoder used during retrieval encoded them independently (fast but less accurate). The cross-encoder reads both as a single input, letting transformer attention directly compare their tokens — far more accurate but slower.
Input — one pair per candidate:
(Query, Chunk C)
(Query, Chunk A)
(Query, Chunk D)
(Query, Chunk B)
Internal token format:
[CLS] How does self-attention work? [SEP]
Self-attention computes relationships between tokens... [SEP]
Output — precision relevance scores:
PairScore
Query + Chunk C0.97
Query + Chunk A0.82
Query + Chunk D0.31
Query + Chunk B0.18
Only the top-scored chunks (e.g. C and A) proceed. Common models: Cohere Rerank, BGE Reranker, Jina Reranker. This app uses the LLM itself for scoring (0–10 per passage) as a reranker approximation.
Step 9 — Context Compression
Before the LLM sees the context, a post-processing pass maximises signal and minimises token waste.
Deduplication
Remove near-duplicate chunks that would waste context window tokens on repeated information.
Summarisation
Extract only the relevant sentences from each chunk rather than passing the full chunk text.
Reordering
Place most-relevant chunks at the start and end of the context — LLMs attend more strongly to these positions.

When to Use It
  • Large, complex documents with mixed content — tables, images, varied formats
  • Users ask vague, multi-part, or domain-specific questions
  • Answer accuracy and reliability are non-negotiable
  • Production systems with real users in high-stakes environments
When Not to Use It
  • Building a quick prototype — each step adds latency and cost
  • Small, clean, well-structured document sets
  • Infrastructure budget is limited
  • Naive RAG has already been tested and proven sufficient
Strengths
  • Handles complex, ambiguous, and multi-part questions effectively
  • Metadata traceability — every answer can be traced to a source, page, and section
  • Reranking ensures only the most relevant context reaches the LLM
  • Answer validation catches hallucinations before they reach the user
Limitations
  • Significantly more complex to build, tune, and maintain
  • Higher latency — multiple LLM calls for rewriting, reranking, and validation
  • Greater infrastructure cost — reranker models, multi-vector storage
  • More failure points — each step can introduce its own errors
Key Concepts
HyDE — Hypothetical Document Embeddings
Instead of embedding the raw question, the LLM first generates a hypothetical answer to the question. That hypothetical is then embedded and used for retrieval. The idea: a fake but plausible answer is semantically closer to the real stored chunks than a short question would be.
Hybrid Search
Combines dense retrieval (vector similarity — good at semantic meaning) with sparse retrieval (keyword/BM25 — good at exact terms, names, codes). Results from both are fused using Reciprocal Rank Fusion (RRF), giving the best of both worlds. This app implements hybrid search in the Advanced and Hybrid RAG endpoints.
Cross-Encoder Reranking
The bi-encoder used during indexing encodes query and chunks independently — fast but less accurate. A cross-encoder takes the query and each candidate chunk together as a single input and outputs a precise relevance score. Slower but far more accurate. Common models: Cohere Rerank, BGE Reranker, Jina Reranker, Cross-Encoder models.
Sliding Window & Parent-Child Chunking
Sliding window chunking uses overlap between consecutive chunks so no idea is cut at a boundary. Parent-child chunking stores small chunks for precise retrieval but also keeps a reference to the larger parent chunk — when a small chunk is retrieved, the full parent is sent to the LLM for richer context.
Reciprocal Rank Fusion (RRF)
A score-free fusion algorithm that merges multiple ranked lists. Each document gets a score of 1 / (k + rank) from each list, and the scores are summed. The constant k=60 is the standard value. This app uses RRF in both the Advanced and Hybrid RAG endpoints.
