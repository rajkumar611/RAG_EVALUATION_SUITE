What is RAG?

RAG stands for Retrieval-Augmented Generation:

Retrieve Find relevant information from your knowledge base
Augment Feed that information to an AI model
Generate AI creates a response using both its training + retrieved knowledge
Without RAG
"What's the latest on AI?"
→ AI uses only training data (may be outdated)
With RAG
"What's the latest on AI?"
→ AI retrieves current documents → Uses them to generate a fresh, accurate answer

Who Decides Which RAG Strategy to Use?

The developer who builds the RAG system decides which strategy to use. When a developer writes the code, they choose every step — how to chunk, how to embed, how to retrieve. This is done through code or configuration files, not through chat commands. If a company uses a platform like Azure or AWS, their technical team sets it through platform settings. If you are an end user of a finished product like a chatbot, the decision was already made for you before you even opened the application. You as an end user have no control over it — the pipeline is fixed at build time, not at runtime.

Strategies Are Not Rigid Boxes

Importantly, it is not always one pure strategy. A developer can mix techniques from both. For example, a system might use fixed-size chunking from Naive RAG, but add hybrid search and re-ranking from Advanced RAG. Another system might use semantic chunking and multiple vectors, but skip post-retrieval compression to keep response times fast. Naive RAG and Advanced RAG are not two rigid boxes — they are a collection of individual techniques, and developers pick and combine them based on their specific needs, budget, and performance requirements.

How Vector Databases Handle Search Behind the Scenes

Once your document chunks are converted into vectors and stored, the vector database does not just sit and wait. It automatically builds internal indexes — data structures that make similarity search fast even across millions of vectors. Two of the most common are:

HNSW Hierarchical Navigable Small World — builds a layered graph of vectors so a search can jump straight to the nearest neighbors without scanning everything.
IVF Inverted File Index — groups similar vectors into clusters so a search only looks inside the most relevant clusters instead of comparing every single vector.

Your RAG pipeline never manages any of this directly. Indexing is handled entirely by the vector database in the background — the pipeline simply stores vectors and queries them.
